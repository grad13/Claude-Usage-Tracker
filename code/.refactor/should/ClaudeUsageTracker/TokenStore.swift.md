# TokenStore.swift

## 問題点

### 1. DBスキーマと TokenRecord モデルの不一致（ハードコード値）

**現状**: queryRecords メソッド（270-275行）で `speed: "standard"` と `webSearchRequests: 0` を固定値としてハードコードしている。DBの token_records テーブルにはこれらのカラムが存在しない。
**本質**: TokenRecord が持つフィールドのうち2つがDBに永続化されておらず、読み出し時にダミー値で埋められている。JSONL元データにこれらの情報が含まれていても、sync時に捨てられてDBには保存されない。将来 speed や webSearchRequests を使った分析・コスト計算が必要になったとき、過去データからは復元できない。
**あるべき姿**: token_records テーブルに speed と web_search_requests カラムを追加し、upsertRecord で実データを保存する。または、TokenRecord の定義からこれらのフィールドを除外して、ストア層とモデル層の一致を保つ。

### 2. sync メソッド内のファイルスキャン責務

**現状**: sync メソッド（50-113行）がディレクトリ走査・ファイルフィルタリング・mod date比較・DB書き込みを一つのメソッドで行っている。約60行のメソッドで、ファイルシステム操作とDB操作が混在している。
**本質**: テスト時にファイルスキャンとDB書き込みを個別に検証できない。ファイルスキャンのロジック変更（例: .jsonl以外の拡張子対応）がDB操作と同じメソッドに閉じ込められている。ただし、現時点では単一ドメイン内に収まっており、緊急度は低い。
**あるべき姿**: ファイルスキャン（対象ファイルの列挙と変更検出）を別メソッドまたは別型に分離し、sync はそれらを組み合わせるオーケストレーションに徹する。
