# build_and_install.py

## 問題点

### 1. register_and_verify に5つの責務が集中

**現状**: `register_and_verify()` (L316-404) が以下を全て担当している:
- バンドルビット検証 (L320-334)
- LaunchServices 登録 + stale 解除 (L337-374)
- デプロイメント検証ゲート (L378)
- データ整合性チェック (L381-395)
- Dock リフレッシュ + アプリ起動 (L398-404)

**本質**: 1関数が「検証」「登録」「データチェック」「起動」を全て行うため、個別のステップをスキップ・リトライ・テストできない。関数名が示す責務（register and verify）を超えてデータ整合性チェックや起動まで含んでいる。

**あるべき姿**: 各責務を独立した関数に分離し、`register_and_verify` はそれらを呼び出すオーケストレーターとする。または `main()` から直接各ステップを呼ぶ。

### 2. DBバックアップ/整合性チェックがデプロイロジックと混在

**現状**: `backup_database()` (L75-101), `rotate_backups()` (L68-72), `check_lost_rows()` (L304-313) がデプロイスクリプト内に直接定義されている。

**本質**: データベース保全はデプロイの「前提条件」であり、ビルド・インストールとは異なるドメイン。現在 `lib/` に `data_protection.py` が既に存在しており、DB関連ユーティリティの置き場所として自然な候補がある。

**あるべき姿**: DB バックアップ・ローテーション・整合性チェックを `lib/` 配下に分離する（例: `lib/db_backup.py`）。デプロイスクリプトはそれをインポートして使う。
