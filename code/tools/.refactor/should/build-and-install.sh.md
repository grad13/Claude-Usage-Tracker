# build-and-install.sh

## 問題点

### 1. 7つの独立した責務がフラットに並んでいる

**現状**: 1つのスクリプトに以下の責務が直列に並んでいる（関数化されていない）:
- データ保護（L19-38: DBバックアップ、settings/cookieスナップショット）
- テストゲート（L40-57: ユニットテスト実行 + テスト後のファイル復元）
- LaunchServices クリーンアップ（L59-68: DerivedData登録解除）
- ビルド（L70-87: xcodebuild）
- アプリ停止・インストール（L89-123: quit/kill、/Applicationsへコピー、ウィジェット検証）
- LaunchServices 登録・検証（L125-158: 登録解除、登録、pluginkit、検証）
- データ整合性チェック（L160-176: デプロイ後のDB行数検証）

**本質**: 全てがトップレベルのインラインコードで、関数に分離されていない。個別の責務をテスト・再利用・差し替えできない。スクリプトの流れを追うには183行を全て読む必要がある。

**あるべき姿**: 各責務を関数に分離し、メインフローは関数呼び出しのパイプラインとして読めるようにする。`lib/data-protection.sh` のように既に外部ライブラリ化のパターンがあるので、同様に `lib/` 配下に切り出すか、少なくともスクリプト内で関数化する。

### 2. LaunchServices DerivedData 登録解除の重複コード

**現状**: 同一のループパターンが2箇所に存在する:
- L62-68（テスト後、ビルド前）
- L127-133（インストール後）

それぞれ `${APP_NAME}-*` と `WeatherCC-*`（旧名）の2ループで、計4ループが2セットある。

**本質**: 同じ処理が2箇所にコピーされているため、変更時に片方を更新し忘れるリスクがある。実際に旧名 `WeatherCC` のクリーンアップが両方に追加されており、今後新たなパターンが増えた場合にも同じ問題が起きる。

**あるべき姿**: `deregister_stale_copies()` のような関数に抽出し、2箇所から呼び出す。

### 3. 旧名(WeatherCC)クリーンアップの散在

**現状**: リネーム移行用のクリーンアップコードが本体の複数箇所に直接埋め込まれている:
- L66-68: DerivedData旧名クリーンアップ（テスト後）
- L131-133: DerivedData旧名クリーンアップ（インストール後、問題2と重複）
- L134-136: Trash内の旧名クリーンアップ

**本質**: 移行用の一時的なコードがデプロイスクリプト本体に分散しており、移行完了後の除去が困難。将来さらにリネームがあれば同様のコードが増殖する。

**あるべき姿**: 旧名クリーンアップを専用関数（例: `deregister_legacy_apps()`）にまとめ、移行完了後に1箇所を削除するだけで済むようにする。
