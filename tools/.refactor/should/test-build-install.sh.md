# test-build-install.sh

## 問題点

### 1. バイナリバックアップロジックの3重複

**現状**: Test 6 (L217-229), Test 7 (L258-270), Test 8 (L294-306) でバージョン取得 + mv のバックアップロジックがほぼ同一のコードとして3回繰り返されている。PlistBuddy でバージョンを取得し、`rm -rf` + `mv` でリネームする13行のブロックが3箇所にコピーされている。
**本質**: テスト対象のロジックがインラインで重複しているため、build-and-install.sh 側のロジックが変わった場合に3箇所を同期させる必要がある。テストの保守コストが高い。
**あるべき姿**: バックアップロジックを関数化し、各テストケースはその関数を呼ぶだけにする（`run_lost_check` と同じパターン）。

### 2. アサーション方式の不統一（assert_eq と手動 if/else の混在）

**現状**: `assert_eq` 関数が定義されている (L46-58) が、Test 5 (L172-198), Test 9 (L327-334), Test 11 (L358-364), Test 12 (L391-398), Test 15 (L452-459) では `if/else` で直接 `PASSED`/`FAILED` をカウントしている。2種類のアサーション方式が混在。
**本質**: 「ファイルが存在するか」「出力に文字列が含まれるか」を検証する汎用アサーション関数がないため、手動で毎回 if/else を書いている。テスト結果の出力フォーマットが不統一になり、新しいテストを追加する際にコピペが発生する。
**あるべき姿**: `assert_file_exists`, `assert_file_not_exists`, `assert_contains` などの汎用アサーション関数を追加し、全テストケースで統一的に使う。

### 3. 5つの独立したテスト対象が1ファイルに混在

**現状**: 1ファイルに以下の5つの独立したテスト対象が含まれている: (1) 行消失検出SQL (Test 1-4), (2) DBバックアップローテーション (Test 5), (3) バイナリバックアップ (Test 6-8), (4) rollback.sh (Test 9-11), (5) data-protection.sh関数 (Test 12-15)。
**本質**: 現時点では469行で管理可能な範囲だが、テスト対象ごとに独立したライフサイクルを持つ。rollback.sh のテストを変更したいときに、データ保護SQLのテストコードを読み飛ばす必要がある。テストの追加に伴い肥大化するリスクがある。
**あるべき姿**: テスト対象ごとにファイルを分割するか、少なくとも共通のテストヘルパー（セットアップ、アサーション関数）を `lib/` に切り出して各テストファイルから `source` する構成にする。
